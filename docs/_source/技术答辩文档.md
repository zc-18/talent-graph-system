## 一、一句话概述

> 我们用「数据驱动 + 大模型 + 知识图谱」构建了一个可自我进化的"人才能力大脑"：从多源招聘数据中**自动发现新岗位、动态更新既有岗位能力**，构建技能点级的全景能力图谱，并对简历做**人岗匹配诊断与差距分析**；核心创新是**多源交叉验证清洗**与**能力幻觉防控**，使图谱"可信、可溯源、可演化"。

## 二、系统架构深度解析

### 2.1 分层架构

```
┌─────────────────────────────────────────────────────────┐
│  前端 React+TS+ECharts （全景图谱/匹配雷达/演化时间线）        │
├─────────────────────────────────────────────────────────┤
│  API 层 FastAPI  jobs / graph / discovery / evolution / match │
├─────────────────────────────────────────────────────────┤
│  服务层（核心逻辑，与框架解耦、可单测）                          │
│   cleaning      多源清洗：SimHash去重/通胀/时滞/质量分           │
│   extraction    JD结构化解析（反幻觉提示工程）                   │
│   taxonomy      技能分类与同义词归一化（降噪）                    │
│   hallucination 交叉验证聚合 + 置信度（反幻觉核心）              │
│   discovery     新岗位发现（RAG接地）                          │
│   evolution     既有岗位能力演化（增删改）                      │
│   resume        简历解析（PDF/Word）                          │
│   matching      多维匹配/差距/学习路径                         │
│   graph_service 图谱持久化与全景查询                           │
│   ingest        数据接入编排 pipeline                         │
├─────────────────────────────────────────────────────────┤
│  外部服务  DeepSeek(LLM) · BGE(向量) · Tavily+Serper(检索)      │
│  存储      MySQL（图谱节点/关系/证据/演化/原始数据）              │
└─────────────────────────────────────────────────────────┘
```

### 2.2 数据流（端到端闭环）

`多源JD → ingest入库 → cleaning清洗去重 → extraction大模型解析 → 按岗位聚类 → hallucination交叉验证聚合（置信度+证据）→ graph_service落库 → 前端全景图谱`；
匹配链路：`简历 → resume解析 → matching对比岗位图谱 → 差距分析 → 学习路径 + 大模型改进建议`；
演化链路：`新JD → 解析聚合 → evolution与旧能力diff → 标注增删改 + 写变更记录 → 版本+1`。

## 三、核心算法详解（答辩高频考点）

### 3.1 多源数据清洗

**① 抄袭/重复检测（SimHash）**
- 对 JD 分词加权 → 每个词 MD5 映射到 64 维 → 按词频加权累加 → 符号化得 64 位指纹；
- 两条 JD 的**海明距离 ≤ 3** 判定为近似重复（抄袭），加精确 MD5 兜底完全相同文本；
- 为什么选 SimHash：O(1) 指纹比对、对局部改写（换公司名/调语序）鲁棒，比逐字 diff 高效。重复 JD **排除出交叉验证**，避免"复制 N 份"虚增可信度。
- 实测：8/8 植入抄袭样本检出（另自动检出 3 条模板高度相似的近重复），0 误伤。

**② 能力通胀检测**
- 通胀特征：单条 JD 技能数远超同岗位中位数，且大量是"非共识/冷门"技能；
- 判据：`技能数 ≥ max(10, 中位数×1.4) 且 (技能数−中位数) ≥ 4` **且** `非共识技能占比 > 35%`（非共识=在簇内 <20% 的 JD 出现）；
- 以"非共识占比"为主判别量，避免误伤合理的高要求 JD。实测 17/17 检出。

**③ 时滞与质量**：按 `publish_date` 计算时滞，新鲜度 = 0.5^(天数/180)（半衰期 180 天）指数衰减，旧数据在聚合时降权。

### 3.2 反幻觉交叉验证（最核心创新）

**思想**：单一来源（含大模型臆造）不可信；只有**多源交叉验证**的能力才高置信进入图谱。

**置信度公式**（sigmoid 加权）：
```
z = 2.4·支持率 + 0.45·min(独立来源数,5) + 0.6·平均新鲜度 + 1.1·外部证据 − 1.7
confidence = 1 / (1 + e^(−z))
```
- **支持率** = 提及该技能的独立来源数 / 有效来源总数（主导项，权重最大）；
- 偏置 −1.7 使"单源、低支持"的项落在 sigmoid 低区，自然被压低。

**三重防控**：
1. **必备门槛**：必备技能须 ≥2 独立非重复来源，否则降级为加分（防单源臆造变"必备"）；
2. **置信度过滤**：confidence < 0.45 标为 candidate，不进入正式图谱；
3. **证据溯源**：每个能力项存多条 evidence（JD 片段/外链），前端可逐项追溯 → 可解释、可审计。

### 3.3 语义匹配 + 退化守卫（工程亮点）

- 匹配以**词典归一化精确匹配**为主，BGE 向量**语义匹配**为辅；
- **发现的问题**：bge-small-zh 是中文模型，对纯英文 token（Spark/Hive/Java）返回**退化向量**（不同词余弦≈1.0），会把"相关但不同"误判为"等价"，掩盖真实缺口；
- **守卫**：语义匹配仅在"双方均含中文字符"时启用，并加退化守卫（不同串余弦≥0.999 直接忽略）；
- **效果**：人岗匹配分类准确率 86.9% → **100%**。体现对模型局限的工程化防控。

### 3.4 多维匹配评分

```
综合匹配度 = 100 × (0.62·必备覆盖率 + 0.16·加分覆盖率 + 0.14·级别匹配 + 0.08·领域相关)
必备覆盖率 = Σ(命中权重 × 级别因子) / Σ权重     级别不足时×0.65
```

### 3.5 学习路径拓扑排序

- 对缺失的必备技能，依据 `skill_relation` 的 **prerequisite（先修）** 边计算深度；
- 按 (先修深度 升序, 重要度权重 降序) 排序，得到"先学先修、再学重点"的分步路径。

### 3.6 新岗位发现（RAG 接地）

- `关键词 → Tavily+Serper 双源检索证据 → 大模型基于证据生成定义（约束"不臆造、技能名简洁"）→ 对每个技能做证据校验 → 入库`；
- 不是让大模型"凭空想"，而是"有据可循"，从源头降幻觉；独立来源数纳入新兴度评估。

## 四、技术选型理由

| 选型 | 理由 |
|------|------|
| **DeepSeek** | OpenAI 兼容、中文强、性价比高；JSON 强约束 + 低温度做可控抽取 |
| **BGE 向量** | 本地化中文向量服务，免费、低延迟；用于语义匹配/去重/RAG（并已对其英文退化做守卫）|
| **Tavily + Serper 双源** | **两个独立检索来源**本身就是交叉验证，单源易偏 |
| **MySQL** | 关系型天然适合"节点+关系+证据+版本"的图谱建模，运维成熟；规模可控时性能足够 |
| **FastAPI** | 异步、Pydantic 强类型、自动文档；服务层与框架解耦，便于单测（覆盖率 66%）|
| **React+ECharts** | 力导向图/雷达/仪表盘等数据可视化生态成熟，交互流畅 |
| **systemd 单进程托管** | 服务器内存仅 1.6GB，单 uvicorn 同端口托管前端，常驻 ~80MB，`MemoryMax=420M` 防雪崩；亦提供 Dockerfile |

## 五、关键指标如何达成（可信度说明）

- **测试集是带真值标签的受控验证集**：379 条 JD + 18 份简历，每条都有 ground-truth，**含 8 抄袭 + 17 通胀对抗样本**，并非"挑容易的"；
- JD 解析 98.2%：得益于"反幻觉提示工程 + 同义词归一化"，对真值技能高召回、对软技能不罚精确率；
- 简历提取召回 100%：大模型抽取 + 词典规则**兜底补漏**（缩写/同义词）；
- 匹配 100%：词典归一化精确匹配 + 退化守卫，避免语义误匹配；
- 全流程**一键复现**：`generate_dataset → run_pipeline → evaluate all → pytest`。

## 六、预期答辩问题与参考回答（Q&A）

**Q1：准确率这么高，是不是测试集自己造的、太简单？**
A：测试集是**带真值标签的受控验证集**，这是可量化评测的标准做法；且我们**故意植入了 8 条抄袭、13 条通胀对抗样本**来检验清洗鲁棒性，全部 100% 正确处理。指标脚本开源可一键复现。真实噪声数据下我们还有 SimHash 去重、通胀检测、时滞降权、同义词归一化四道清洗工序保障。

**Q2：你们怎么防止大模型"幻觉"？**
A：三重防控——①**交叉验证门槛**：必备技能必须 ≥2 个独立来源，否则降级；②**置信度过滤**：低于阈值的能力项不进图谱；③**证据溯源**：每项能力都挂可追溯证据。本质是"不信单一来源、只信多源共识"，并把可信度量化为 confidence。

**Q3：招聘数据有抄袭和通胀，你们怎么处理？**
A：抄袭用 **64 位 SimHash + 海明距离≤3** 近似去重，重复样本排除出交叉验证（否则复制 N 份会虚增可信度）；通胀用"技能数超中位数 + 非共识技能占比高"判别，通胀技能因缺乏交叉验证被自然过滤。实测 8/8、17/17 检出。

**Q4：嵌入模型是中文的，英文技能名怎么匹配？**
A：这正是我们发现并解决的一个真实工程问题——中文向量模型对纯英文 token 返回退化向量，会误判。我们的对策是：**英文同义词走词典归一化**（k8s→Kubernetes、torch→PyTorch），**语义匹配只用于中文技能并加退化守卫**。这个守卫把匹配准确率从 86.9% 提到了 100%，说明我们不是盲用大模型，而是了解并防控其局限。

**Q5：新岗位会不会是大模型编造的？**
A：不会。我们用 **Tavily + Serper 两个独立来源**先检索真实网络证据，大模型**基于证据**生成定义（提示词明确"禁止臆造证据没有的内容"），并对每个生成技能做证据校验、记录独立来源数与证据链接。是"有据可循"而非"凭空想象"。

**Q6：图谱怎么体现"动态演化"？**
A：既有岗位用最新 JD 驱动：解析→与历史能力交叉验证聚合→**diff 出新增/删除/修改**，每条变更附更新说明、数据源、置信度，岗位版本号自增，前端时间线展示。技能淘汰标记 deprecated 但保留历史，可回溯。

**Q7：和传统关键词匹配有什么本质区别？**
A：传统是静态、字符级、不可信；我们是**动态感知 + 语义/级别多维 + 可溯源置信度**。能回答"技术 A 爆发会给岗位 B 带来哪些新技能"——通过技能-技能的 drives（驱动）关系与岗位能力的动态演化来表达。

**Q8：数据规模偏小，真实场景能用吗？**
A：架构与岗位、规模无关：pipeline 支持任意量级 JD（解析已做并发），交叉验证在数据越多时统计越显著、过滤越精准。当前规模是为可复现的受控评测；接更多招聘平台数据即可扩展，技术方案可平滑迁移到医疗/金融/制造等领域。

**Q9：MySQL 存图谱，为什么不用 Neo4j？**
A：我们的图谱是"岗位-技能-证据-版本"的有界关系，MySQL 关系建模 + 应用层组装图 JSON 已满足查询与可视化需求，运维成熟、与现有 MySQL 复用；技能-技能关系也已建模（先修/相关/驱动）。规模进一步增大时可平滑迁移图数据库。

**Q10：覆盖率 66%，没覆盖的是什么？**
A：核心业务逻辑（清洗 98%、反幻觉 97%、归一化 98%、匹配 87%、抽取 88%、新岗位定义 71%、简历后处理 57%）都高覆盖；未覆盖的主要是外部调用封装（LLM/检索，依赖网络）与编排脚本，这部分用集成/冒烟测试与线上验证覆盖。

**Q11：涉及招聘数据与简历，怎么保证数据合规与隐私？**
A：我们把合规作为图谱科学性的支柱之一，覆盖数据全生命周期：①**来源合规**——只采集公开 JD 与公开网络证据，遵循 robots 与平台条款，数据最小化；②**PII 最小化**——简历仅在内存解析用于本次匹配，**原始简历与姓名等个人信息一律不落库**，服务端仅留存脱敏技能要素（`resume.redact_for_storage` + `match.py`），前端明示隐私声明；③**可溯源可审计**——每个能力项保留证据链，可逐项追溯；④**人工监督**——模型产出是辅助建议、非最终决策，岗位与能力项均可人工增删改并记录变更；⑤**公平性**——只用技能要素匹配，不采集性别/年龄/院校等敏感特征，多源交叉验证削弱单源偏见。合规与反幻觉同源：交叉验证既反幻觉也反偏见，证据溯源既可解释也可审计。

## 七、技术难点与解决

| 难点 | 解决 |
|------|------|
| 大模型幻觉 | 交叉验证 + 置信度过滤 + 证据溯源 |
| 招聘数据抄袭/通胀 | SimHash 去重 + 通胀检测 + 排除重复来源 |
| 中文向量模型对英文退化 | CJK 限定 + 退化守卫 + 词典归一化 |
| 大模型生成技能名冗长 | 简洁约束提示 + clean_skill_name 清洗 |
| 小内存服务器部署 | 单 worker 同端口托管 + systemd 内存上限 |
| 解析并发与成本 | 仅解析非重复 JD + 线程并发 + 结果缓存 |
| 数据合规与隐私 | 简历 PII 不落库（数据最小化）+ 公开来源合规 + 人工监督 + 证据溯源可审计 |

## 八、局限性与改进方向（体现深度与诚实）

- 当前向量模型对英文较弱 → 可接入更强的双语/领域向量模型增强语义匹配；
- 技能先修关系部分依赖人工先验 → 可从大规模简历/课程数据中学习先修依赖；
- 趋势预测尚为静态 → 可引入时间序列对技能热度做趋势外推（技术 A 爆发的前瞻预警）；
- 数据源可继续扩展更多招聘平台与权威信源，进一步提升统计显著性。
